MVP Requirements

Based on talking with users across the Ads teams and with the product team, we came up with design requirements that guided our creation of an ideal workflow for the Video Ads Module MVP interface. From these design requirements, we were able to create wireframes and high-fidelity designs for the MVP.

Product Purpose

Since MPS currently only services display ads, NBCU needed a system that could handle video ads. This new Video Ads Module is a rules engine that will allow the Ads team to control video ads configuration from MPS, which will serve as the front-end. By introducing this new level of functionality to MPS, NBCU will be able to reduce costs, enable advanced ad workflows, and allow for multiple ad experiences within a single application.

With how video ads are served today, a player makes an ad request to Freewheel that contains a lot of information about that stream. The Video Ads Module will allow Ads teams to manage the structure of that ad request quickly and on the fly. Today, if the Ad Integration and Architecture wants to modify a parameter in the ad request, they need the Ad Integration and Architecture team to log a ticket with a developer, and the developer needs to make a code change to change the value. The code change then has to be tested, deployed, and tested in production as well. To change five letters in an ad tag can take up to two weeks.

The Video Ads Module will give the Ad Integration and Architecture team the ability to make changes on their own, which will be faster and free from app updates or releases. Ultimately, the Video Ads Module will allow for more flexibility and scalability in serving up video ads.

MVP Design Requirements

These features required in the MVP are ranked in order from highest priority to lowest priority.

  1. Hierarchy
    • Users need to be able to access different player instances within each brand, each platform, and various modes within those instances to configure ad experiences.
    • Users must be able to specify the brand, the platform, and then the mode.
      • The number of brands in the Ads Module should expand to 10-15.
      • The minimum number of platforms is one, but there is no set maximum. Ideally, there would be 7-8 platforms for a brand.
      • The 2 different modes supported in the MVP are on-demand (VOD) and live.
  2. Parameters
    • Different parameters will be available depending on features being turned on or based on context.
      • The majority of parameters are not tied to a feature.
      • There can be up to 30 parameters and 22-24 of them are a part of the base set that is not feature-dependent.
    • Users need the ability to change parameters that help build the ad tag. This will prevent the Ad Integration and Architecture team from having to file tickets for minute parameter changes that take two weeks.
    • There are two types of parameter values: single and multiple.
      • Less than 10% of parameters would involve multiple value entry.
      • A majority of these parameter fields will be text input fields and users can input tokens that act as placeholder values (i.e., vod_{DEVICE} → vod_iphone).

  3. A/B Test Mapping
    • Users will be able to configure A/B testing per brand within the Ads Module. This configuration will apply to each device per brand.
    • A/B testing involves mapping the audience ID and the variant ID (assigned by a 3rd party user audience tracking system) to the experience template ID (which is configured in Freewheel).
    • Users will also be able to input a description to name the type of mapping specified by those parameters.
  4. Ability to Switch Instances within MPS
    • Since the MVP will be incorporated into MPS, there needs to be a way for a user to return to MPS to access other instances. Users will access the MVP through the MPS UI.
  5. Ads Module Environment
    • Users need to be able to selecting their access environment in the Ads Module. Environments include: QA, Staging, and Production.
    • After users select the environment they want to access, they will input the corresponding network ID parameter that applies to all brands, platforms, and modes.
